## 2. 課題（Problem）

### 2.1 解決したい課題

**主要課題（1つだけ）**:
```
セミナー・イベント運営は、関わるプレイヤーが多い（主催者・会場・配信業者・企画代行・登壇者・営業マーケ・参加者・外注業者）にも関わらず、
情報がメール・Excel・電話・チャットにバラバラに散在しており、全員が同じ情報の上で動けていない。
結果として、タスクの抜け漏れ・調整の手戻り・属人化・機会損失が慢性的に発生している。
```

**関連する課題**:
1. 会場運営者が「箱だけ」の提供に留まり、配信付きワンストップサービスとして差別化できていない（JV機会の逸失）
2. イベント終了後の参加者データ・アンケート・質問ログが活用されず、リード商談化・次回企画・追加提案に繋がらない
3. イベント準備の大半が担当者の経験と手作業に依存しており、品質のバラつきとスケール限界がある

### 2.2 課題の深掘り

| 質問 | 回答 |
|------|------|
| この課題は誰が持っている？ | セミナー・イベントに関わる全プレイヤー。特にセミナー主催者（企画〜報告の全工程を担う）と会場運営者（営業〜当日オペの全工程を担う）に負荷が集中 |
| どんな状況で発生する？ | イベントの企画・準備・当日運営・事後フォローの全フェーズで常時発生 |
| どのくらいの頻度で発生する？ | イベントを実施するたびに毎回。企業のセミナー開催頻度は月1〜5回程度 |
| 発生すると何が困る？ | タスク漏れによるトラブル、調整の手戻りによる時間ロス、機会損失（リード未活用・リピート逃し）、担当者の疲弊 |
| 今はどう対処している？ | メール・Excel・電話・チャット・紙のチェックリスト・担当者の記憶と経験 |
| 対処にいくらかかっている？（時間・お金） | 1イベントあたり企画〜報告で延べ40〜100時間（関係者合計）。会場営業は1案件60分の対応時間 |

### 2.3 課題の検証状況

| 検証方法 | 実施 | 結果 |
|---------|------|------|
| 自分自身が課題を経験 | ☑ | 配信プラスとして多数のイベント配信案件で、情報散在・引き継ぎミス・追加提案漏れを経験 |
| 身近な人にヒアリング（5人以上） | ☑ | ビジョンセンター（JV先）との協議で、会場側のペイン（配信案件の営業困難・調整負荷）を確認 |
| SNS/コミュニティでの反応確認 | ☐ | 未実施 |
| 既存サービスのレビュー分析 | ☐ | 未実施（競合分析は今後の課題） |
| 検索ボリューム確認 | ☐ | 未実施 |

## 3. ソリューション（Solution）

### 3.1 提供する解決策

```
イベント運営に関わる全プレイヤーが1つのWebシステムにログインし、
日程・タスク・締め切り・必要情報を一元管理する。
さらにAIアシスタント（HUBコンシェルジュ）に「聞く・頼む」ことで、
企画書生成・タスク展開・リマインド・文面作成・レポート生成などの業務をAIが代行し、
運営の手間を大幅に削減する。
```

### 3.2 主要機能（MVP）

| # | 機能 | 課題との関連 | 優先度 |
|---|------|-------------|-------|
| 1 | イベント企画・予約 AIアシスト | 企画のゼロベース作業削減。会場営業の即時見積り（JV基盤） | P0 |
| 2 | タスク・締め切り一元管理（マスタースケジュール） | 全プレイヤーのタスク・締め切りの抜け漏れ防止 | P0 |
| 3 | ステークホルダー・ワークフロー管理 | 登壇者・関係者との調整工数削減 | P0 |
| 4 | 参加者レイヤー（ポータル＋QR＋AI FAQ） | 参加者への情報提供一元化、会場受付負荷軽減 | P0 |
| 5 | 事後レポート＆フォローアップ | イベント後のデータ活用・追加提案・リード商談化 | P0 |

**注意**: MVP は 5 機能に絞る（＋共通基盤のAIアシスタントUI）

### 3.3 将来の機能（Phase 2以降）

| # | 機能 | 実装時期 |
|---|------|---------|
| 1 | MA/CRM連携（双方向リアルタイム同期） | Phase 2 |
| 2 | オンライン視聴ログ分析（視聴時間・チャプター離脱） | Phase 2 |
| 3 | マルチ会場チェーン対応・OEM提供 | Phase 2 |
| 4 | ネイティブモバイルアプリ | Phase 2 |

| 6 | 年間イベント計画・予算管理機能 | Phase 3 |
| 7 | 多言語対応（管理画面・参加者ポータル） | Phase 3 |

## 4. ターゲットユーザー（Target User）

### 4.1 プライマリターゲット

| 項目 | 内容 |
|------|------|
| 誰？ | セミナー・イベントの主催者（企業のマーケ部・人事部・広報部など） |
| 年齢層 | 30〜50代 |
| 職業/役割 | マーケティング部門の担当者〜マネージャー |
| 課題の深刻度 | 高（毎回のイベントで発生し、担当者の業務時間の大部分を占める） |
| 支払い意思 | 高（イベント予算の一部として計上可能。現状の外注費・人件費を考えれば十分にROIあり） |
| リーチしやすさ | 高（JV先の会場チェーン経由で直接アクセス可能） |

### 4.2 セカンダリターゲット

| 項目 | 内容 |
|------|------|
| 誰？ | 貸し会議室運営事業者（JV先） |
| プライマリとの違い | 自社がシステムを利用する側でもあり、同時に顧客（主催者）への窓口でもある。ビジネスパートナーとしてHUBの導入・運用を担う |

### 4.3 ターゲットではない人

| 対象外 | 理由 |
|-------|------|
| 個人主催の小規模イベント（10名未満） | システムの価値が薄い。手作業で十分回せる規模 |
| 大規模フェス・音楽ライブ | 運営構造が根本的に異なり、専門システムが別途存在する |
| 純粋なオンラインウェビナーのみ | Zoom Webinar等の既存ツールで完結する場合が多い |

## 5. 差別化（Differentiation）

### 5.1 なぜあなたが作るのか？

| 質問 | 回答 |
|------|------|
| この領域の経験・知識 | 配信プラスとして年間数百件のイベント配信・撮影を実施。現場の課題を熟知 |
| ユニークな視点・アプローチ | 「配信業者」と「会場」のJVという独自のポジション。両方の視点から課題を把握している |
| 他の人にない強み | ビジョンセンターとのJV合意があり、実際の会場チェーンでパイロット導入できる |
| なぜ今このタイミング？ | ハイブリッド・配信付きイベントが標準化しつつある中、「箱＋配信」をワンストップで提供できる仕組みが求められている |

### 5.2 競合との違い

> 詳細は `docs/idea/COMPETITOR_ANALYSIS.md` を参照

| 比較軸 | EventHub（国内シェアNo.1） | Peatix / EventRegist | 会場予約システム | **Haishin+ HUB** |
|-------|---------------------------|----------------------|----------------|-------------------|
| ポジション | イベントマーケティングPF | 集客・チケット販売 | 会場の空き管理・予約 | **イベント運営OS** |
| ターゲット | 主催者（マーケ部門） | 主催者（集客側） | 会場運営者 | **主催者＋会場＋配信＋全関係者** |
| AI活用 | なし（2026/2月時点） | なし | なし | **全工程でAIアシスタント稼働** |
| 配信 | プラットフォーム内蔵 | なし | なし | **配信プラスがJVで実務対応** |
| MA/SFA連携 | ◎（Salesforce, Marketo, HubSpot） | △ | なし | △（Phase 2） |
| 会場JV基盤 | なし | なし | 予約のみ | **◎（ワンストップ見積り）** |
| 全プレイヤー管理 | △（主催者中心） | × | × | **◎（8ロール対応）** |
| タスク自動展開 | なし | なし | なし | **◎（マスタースケジュール）** |
| **差別化** | マーケ分析・リード獲得 | 集客ネットワーク | 予約管理 | **全プレイヤー横断＋AI＋JV基盤** |

### 5.3 参入障壁（Moat）

```
競合が真似しにくい理由:

1. 会場チェーン（ビジョンセンター）とのJV関係: 実際の会場データ・運営ノウハウへのアクセス。
   他社が同等の連携を構築するには会場チェーンとの信頼関係構築から始める必要がある。

2. 配信業者としての実務知識: 配信・撮影の現場を知っているからこそ設計できる
   「配信込みワンストップ」のプロセス設計。SaaS企業には持てないドメイン知識。

3. ネットワーク効果: 全プレイヤーが同一システム上にいることで、
   関係者が増えるほどシステムの価値が上がる構造。
```

## 7. 市場規模（Market Size）

### 7.1 TAM/SAM/SOM

| 指標 | 規模 | 算出根拠 |
|------|------|---------|
| TAM（全体市場） | 推定数百億円規模 | 日本のイベント・MICE産業（セミナー・カンファレンス・展示会）の運営管理ツール市場 |
| SAM（対象市場） | 推定数十億円規模 | 貸し会議室でのセミナー・説明会市場（ハイブリッド配信付き案件） |
| SOM（獲得可能市場） | 推定数億円規模 | ビジョンセンター＋同業会場チェーン数社への導入 |

### 7.2 市場トレンド

| トレンド | 影響（追い風/向かい風） |
|---------|---------------------|
| ハイブリッドイベントの標準化 | 追い風：配信付き会場への需要増 |
| 生成AIの普及 | 追い風：AI活用への抵抗感低下 |
| 人手不足（イベント運営人材） | 追い風：省力化ツールへの需要増 |
| 景気変動によるイベント予算削減 | 向かい風：イベント件数減少の可能性 |

## 8. リスクと前提

### 8.1 主要なリスク

| リスク | 影響度 | 対策 |
|-------|-------|------|
| JV先の利用定着が進まない | 高 | MVPで「営業即時見積り」「受付負荷軽減」の体感価値を優先開発 |
| AI生成品質が期待に達しない | 高 | 人間レビューフローの組み込み、プロンプト継続改善 |
| 個人情報漏洩 | 高 | マルチテナント分離、暗号化、PII対策、監査ログ |
| LLM APIコスト増大 | 中 | キャッシュ戦略、利用量モニタリング |
| 競合の参入 | 中 | JV関係＋ドメイン知識による参入障壁を活用 |

### 8.2 検証すべき前提

| 前提 | 検証方法 | 検証時期 |
|------|---------|---------|
| 会場営業はHUBで配信パッケージを売れるか | パイロット導入での受注件数計測 | Phase 1.5 |
| 主催者はAI生成の企画書・メール文面を使うか | パイロットでの利用率＋フィードバック | Phase 1.5 |
| 参加者ポータル+AIで受付質問が減るか | 問い合わせ数のBefore/After計測 | Phase 1.5 |
| LLM APIコストが収益モデルに収まるか | パイロット期間のコスト実績 | Phase 1.5 |

## 9. 成功の定義

### 9.1 6ヶ月後の成功指標

| 指標 | 目標値 |
|------|-------|
| パイロット導入拠点数 | 1拠点（ビジョンセンター1会場） |
| パイロット期間中のイベント管理件数 | 20件以上 |
| 配信付き案件化率（パイロット拠点） | Before比 2倍以上 |
| 主催者のHUB利用継続率 | 80%以上 |
| 参加者ポータルのNPS | +30以上 |

### 9.2 「これが起きたら失敗」

- JV先（ビジョンセンター）がパイロット期間後に導入を辞退する
- 主催者が「Excelの方が楽」と感じてHUBを使わなくなる
- AI生成コンテンツの修正工数が、手作業と変わらないレベル
