## §1. プロダクト概要 [CORE]

### 1.1 一言で何か（エレベーターピッチ）

```
セミナー・イベントの主催者・会場・配信業者・企画代行・登壇者・営業マーケ・参加者 向けの
イベント運営OS で、
関係者間の情報分断・タスク管理の煩雑さ・属人化した調整業務 を解決する。
メール・Excel・電話によるバラバラな運営 とは異なり、
全プレイヤーが1つのシステムに集まり、AIに「聞く・頼む」だけで運営が回る を提供する。
```

### 1.2 背景・課題

現状の課題を具体的に記述：

- **情報の分断**: セミナー・イベント運営に関わるプレイヤー（主催・会場・配信・企画代行・登壇者・営業マーケ・参加者・外注業者）の情報がメール・Excel・電話・チャットにバラバラに散在し、最新情報が誰も把握できない
- **タスク・締め切りの抜け漏れ**: 複数プレイヤーにまたがるタスクがどこにも一元管理されておらず、締め切りの見落とし・重複作業・手戻りが頻発する
- **属人化した調整業務**: 会場手配・配信構成・登壇者調整・告知文作成・当日オペ・事後報告のすべてが担当者の経験と手作業に依存し、スケールしない
- **JV先（貸し会議室）の商機逸失**: 配信・ハイブリッド案件が増加しているにも関わらず、会場運営者は「箱だけ」の提供に留まり、配信付きパッケージを自社サービスとして販売できていない
- **イベント後のデータ活用不足**: 参加者データ・アンケート・質問ログが散在し、リード活用・次回企画・追加提案に繋がらない

### 1.3 解決策

本プロダクトがどのように課題を解決するか：

- **ワンプラットフォーム**: 全プレイヤーが同一システムにログインし、イベント単位で日程・タスク・締め切り・必要情報を一元管理する（MUST）
- **AIアシスタント（HUBコンシェルジュ）**: 「画面を探す」のではなく「AIに聞く・AIに頼む」ことで、企画書生成・タスク管理・リマインド・文面作成・レポート生成などの業務をAIが代行する（MUST）
- **マスタースケジュールの自動展開**: イベント種別に応じた標準タスクテンプレートが自動展開され、各ロールにタスクが自動アサインされる（MUST）
- **JV基盤としての会場価値向上**: 会場営業が「配信込みワンストップ会場」として即時見積り・パッケージ提案を行える仕組みを提供する（SHOULD）
- **事後データの自動集約・レポート生成**: 参加者数・アンケート・質問ログを自動集計し、サマリーレポート・フォローアップ施策・追加提案をAIが生成する（SHOULD）

### 3.2 ユーザーストーリー

```
UC-001:
As a セミナー主催者,
I want to 目的・ターゲット・規模・予算・日程候補を入力するだけで、会場候補・配信構成案・概算見積り・稟議用企画書ドラフトを自動生成してほしい,
so that 企画のゼロベース作業を大幅に削減し、意思決定に集中できる.
```

```
UC-002:
As a イベント関係者（主催・会場・配信・企画代行）,
I want to イベント日を基準にした標準タスクテンプレートが自動展開され、自分のロールに関係するタスクだけがダッシュボードに表示されてほしい,
so that タスクの抜け漏れ・重複を防ぎ、全プレイヤーの進捗が見える化される.
```

```
UC-003:
As a セミナー主催者,
I want to 登壇者情報フォームが自動生成されURLを送るだけで情報を回収でき、AIが紹介文を自動生成してほしい,
so that 登壇者とのやり取りにかかる工数を最小化できる.
```

```
UC-004:
As a イベント参加者,
I want to 1つのポータルでイベント情報を全て確認でき、当日はQRでチェックイン、分からないことはAIに聞ける,
so that メールやPDFを探す手間なく、迷わずイベントに参加できる.
```

```
UC-005:
As a セミナー主催者,
I want to イベント終了後に参加者数・アンケート・質問の自動集計レポートが生成されてほしい,
so that 事後報告の作成工数を削減し、次回企画への示唆を得られる.
```

```
UC-006:
As a 会場営業スタッフ,
I want to 利用人数・形式・配信有無を入力するだけで、配信込みパッケージの概算見積りが即時提示されてほしい,
so that 配信の専門知識なしでもワンストップ会場として受注できる.
```

## §4. スコープ定義 [CORE]

### 4.1 Goals（やること）

- [x] 全プレイヤーが1つのシステムでイベント情報を一元管理できる（MUST）
- [x] AIアシスタントによるイベント企画・見積り・企画書の自動生成（MUST）
- [x] イベント種別ごとのマスタースケジュール（タスク・締め切り）の自動展開（MUST）
- [x] ロール別ビュー（主催者・会場・配信・企画代行・登壇者・営業マーケ・参加者）（MUST）
- [x] 登壇者情報フォームの自動生成・回収・紹介文AI生成（SHOULD）
- [x] 参加者ポータル（情報一元表示・QRチェックイン・AI FAQ）（SHOULD）
- [x] イベントサマリーレポートの自動生成（SHOULD）
- [x] 会場チェーンごとのマルチテナント対応（MUST）

### 4.2 Non-Goals（やらないこと）

明示的にスコープ外とするもの：

- 配信そのものの実行（OBS・vMixなどの配信ソフトウェア機能）
- 動画編集・制作機能（HUBは提案導線のみ）
- リアルタイムの配信モニタリング（視聴者数の簡易表示は含むが、詳細分析は対象外）
- ネイティブモバイルアプリ（Phase 1はWebブラウザ対応）
- 決済・請求書発行機能（既存の請求フローを利用）
- CRM / MA ツールとのリアルタイム双方向連携（Phase 1はエクスポートまで）

### 4.3 将来の検討事項（Future Considerations）

今回は対象外だが、将来検討するもの：

- セミナー以外のイベント（展示会・式典・社内行事）への拡張
- MAツール・CRMとの双方向連携（リード管理・スコアリング）
- オンライン視聴ログ（視聴時間・チャプターごとの離脱分析）の取り込み
- 年間のイベント計画・予算管理・プラン提案機能
- マルチ会場チェーン対応・OEM提供
- ネイティブモバイルアプリ
- 多言語対応（管理画面・参加者ポータル）

## §6. 制約条件 [CONTRACT]

### 6.1 技術制約

- AI部分は外部LLM（Claude / GPT 等）＋ツール呼び出し（Function Calling / Tool Use）によるエージェント構成
- Webアプリケーション（PC／スマホブラウザ対応）。Phase 1ではネイティブアプリ非対応
- マルチテナント設計（会場チェーンごとにデータ分離）

### 6.2 ビジネス制約

| 制約種別 | 内容 |
|---------|------|
| 納期 | Phase 1（MVP）: JV先（ビジョンセンター）へのデモ可能な状態を優先 |
| 予算 | 初期開発はスモールチーム（AI活用による開発効率化を前提） |
| 人員 | 開発: Claude Code ベースの自動AI開発フレームワーク活用。実務ドメイン知識: 配信プラスチーム |
| 法規制 | 個人情報保護法対応（参加者データの取り扱い）、Cookie同意 |

### 6.3 運用制約

- JV先（会場チェーン）のITリテラシーに配慮したUI/UXが必要
- 会場現場のタブレットでの利用を想定（レスポンシブ対応必須）
- 既存の会場予約システムとの連携（API or CSV取り込み）が将来必要

## §8. リスク [DETAIL]

| リスク | 影響度 | 発生確率 | 対策 |
|-------|-------|---------|------|
| JV先（会場チェーン）のHUB利用定着が進まない | High | Mid | MVPで「会場営業の即時見積り」「受付負荷軽減」を体感できるデモを優先開発 |
| AI生成コンテンツの品質が利用者の期待に達しない | High | Mid | プロンプトの継続改善、人間によるレビュー・修正フローを組み込む |
| 個人情報漏洩（参加者データ） | High | Low | マルチテナント分離、アクセス制御、暗号化、監査ログの徹底 |
| 外部LLM APIのコスト増大 | Mid | Mid | キャッシュ戦略、利用量モニタリング、コスト上限アラート |
| 会場予約システムとの連携が困難 | Mid | Mid | Phase 1はCSV取り込みから開始し、段階的にAPI連携へ移行 |
| マルチテナント設計の複雑化 | Mid | Low | 初期からテナント分離を設計に組み込む |

| 検証対象 | 方法 | 合格基準 |
|---------|------|---------|
